# Particle Event System

## Purpose

The particle event system lets one particle module publish runtime facts such as spawn, death, burst, and future collision events, while another module reacts to those facts later in the same emitter tick.

This mirrors the Unreal Cascade split:

- `UParticleModuleEventGenerator` decides which events are emitted.
- `FParticleEventData` carries event payload data.
- `UParticleModuleEventReceiverBase` filters events by type/name.
- Receiver modules such as `UParticleModuleEventReceiverSpawn` and `UParticleModuleEventReceiverKillParticles` perform the reaction.
- `UParticleModuleEventSendToGame` is the extension point for forwarding particle events to game code.

## Runtime Flow

1. `FParticleEmitterInstance::Tick` clears its per-tick event queue.
2. Spawn, burst, and death moments notify enabled event generator modules.
3. A generator checks its `FParticleEventGenerateInfo` entries and queues matching `FParticleEventData`.
4. Queued events are also copied to `UParticleSystemComponent::ParticleEvents` for external inspection.
5. Receiver modules run during the update phase and consume a snapshot of the emitter event queue.

This means events generated by normal spawn/death/burst work in the same frame. Events produced by receiver-triggered spawns are queued, but a receiver does not recursively consume the events it generated during its own update call.

## Added Types

- `EParticleEventType`
  - `Any`
  - `Spawn`
  - `Death`
  - `Collision`
  - `Burst`
  - `Kismet`

- `FParticleEventData`
  - event type/name
  - emitter time
  - particle normalized time
  - particle count
  - location, velocity, direction, normal
  - bone name placeholder for collision-style payloads

- `FParticleEventGenerateInfo`
  - generator event type
  - frequency and particle frequency gates
  - first/last-only flags
  - reflected-impact/orbit flags reserved for collision parity
  - custom event name

## Modules

### `UParticleModuleEventGenerator`

This is a spawn module so it can observe newly spawned particles through the existing spawn module pass. It is also registered in `UParticleLODLevel::EventGeneratorModules`, allowing the emitter instance to notify it when particles die or bursts occur.

Collision support is connected through `UParticleModuleCollision`, which calls `HandleParticleCollision` when a particle raycast hits a configured collision object type.

### `UParticleModuleEventReceiverBase`

This is an update module. It filters queued events by `EventGeneratorType` and `EventName`. Empty/default event names match any name.

### `UParticleModuleEventReceiverSpawn`

Spawns particles in response to matching events.

- `SpawnCount` is a distribution, so it can later be constant, random, curve, or parameter-driven.
- `bUseParticleTime` evaluates the spawn count from event particle time instead of emitter time.
- `bUsePSysLocation` spawns at the component/emitter origin instead of the event location.
- `bInheritVelocity` applies event velocity to spawned particles, scaled by `InheritVelocityScale`.

### `UParticleModuleEventReceiverKillParticles`

Kills all active particles in the receiver emitter when a matching event is seen. If `bStopSpawning` is enabled, it suppresses later spawn calls on that emitter instance.

### `UParticleModuleEventSendToGame`

This exists as the game-facing extension point. The base class is currently a no-op; custom subclasses can override `DoEvent`.

## Current Limitations

- Collision event generation is connected for the CPU collision module. The current implementation supports one collision object type per module; Unreal's full `CollisionTypes` array and GPU collision path remain future work.
- `bUseReflectedImpactVector` and `bUseOrbitOffset` are stored for Unreal parity, but their behavior belongs to collision/orbit integration.
- `UParticleModuleEventSendToGame` exists and serializes as an instanced object list on the generator, but no gameplay delegate layer is wired yet.
